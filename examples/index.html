The expedition planning domain.

A complete governed domain: 6 capabilities, 5 events, 1 workflow. Everything is contract-defined, deterministically executed, and fully traceable. Runnable today with v0.7.0.

Runnable · v0.7.0 6 capabilities 5 events 1 workflow
Domain overview

6 capabilities. 5 events. 1 workflow.

The expedition planning domain models a real multi-step business process. Each step is an atomic governed capability. Each transition is a typed event. The workflow composes all five into a deterministic graph.

interpret-expedition-intent
Parse and structure planning goals from natural language input
emits → expedition-intent-interpreted
capture-expedition-objective
Normalize destination, timing, and team preference data
emits → expedition-objective-captured
assess-conditions-summary
Evaluate weather, terrain, and logistics constraints
emits → conditions-summary-assessed
validate-team-readiness
Check team skill levels and equipment requirements
emits → team-readiness-validated
assemble-expedition-plan
Compose the final governed plan from all validated inputs
emits → expedition-plan-assembled
plan-expedition (workflow orchestrator)
Deterministic graph traversal across all 5 atomic capabilities above. Same inputs always produce the same path.
Step 1: Inspect the bundle

Verify the registry bundle.

Before running anything, inspect the bundle to confirm all capabilities, events, and workflows are registered correctly.

traverse-cli · bundle inspect
$ cargo run -p traverse-cli -- bundle inspect \

Step 2: Run the browser demo

Start two terminals.

The browser adapter listens on port 4174. The React demo connects to it on port 4173.

Terminal 1: browser adapter
$ cargo run -p traverse-cli -- \
browser-adapter serve \
--bind 127.0.0.1:4174

adapter listening on :4174
Terminal 2: React demo
$ node apps/react-demo/server.mjs \
--adapter http://127.0.0.1:4174 \
--port 4173

demo ready → http://127.0.0.1:4173

Contract anatomy

What a capability contract looks like.

Every capability in the expedition domain is backed by a machine-readable contract. This is what governs discovery, validation, placement, and execution.

examples/expedition/capabilities/plan-expedition/contract.json (excerpt)
{
  "id": "expedition.planning.plan-expedition",
  "version": "1.0.0",
  "lifecycle": "active",
  "execution": {
    "binary_format": "wasm32-wasi",
    "entrypoint": "plan_expedition",
    "preferred_targets": ["local"],
    "constraints": {
      "host_api_access": "none",
      "network_access": "forbidden",
      "filesystem_access": "none"
    }
  },
  "events": {
    "emits": [
      "expedition.planning.expedition-plan-assembled"
    ]
  }
}
